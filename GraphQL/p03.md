
# Chapter 3. Why Companies Adopt GraphQL

## 1. Introduction  
Companies use GraphQL because it makes data fetching flexible and efficient.  
Modern apps have many screens, widgets, and devices that all need different sets of data.  
REST often becomes hard to maintain because every new screen may need new endpoints.  
GraphQL fixes this by giving one endpoint that can serve many different frontends.  
This helps teams move faster with fewer backend changes.

---

## 2. Why This Matters  
- Mobile apps need smaller, optimized responses.  
- Dashboards need many small data pieces in one shot.  
- Multiple frontends (web, mobile, admin panels) need different fields.  
- Backend doesn’t break each time the UI team adds a new screen.

---

## 3. Real-Life Example (Courier Service Story)  
Imagine a courier company with different staff roles:

- Delivery boys need only **address + package size**.  
- The billing desk needs **customer info + payment details**.  
- The warehouse needs **package weight + storage location**.

REST is like giving all staff the same giant paper form.  
Everyone gets extra fields they don’t need.

GraphQL is like giving each role a **custom mini-form**—each team asks for the exact fields needed for their job.

---

## 4. Practical Code Illustration

### A. REST (Mobile App Problem Example)  
Mobile needs only the customer name:

GET /customer/10 → returns name, phone, email, address, metadata, timestamps, …

````

Heavy and slow for mobile networks.

### B. GraphQL (Same Scenario)  
```graphql
query {
  customer(id: 10) {
    name
  }
}
````

Fast and tiny.
